Migrating a fintech platform to EKS without a maintenance window

Thirty-odd services, one ageing EC2 fleet, a payments business that couldn’t afford a single planned downtime slot. Here’s how we got off the old cluster in eight weeks, what the Terraform layout looked like, and the two things that almost sank us.

TL;DR

We migrated a fintech transaction platform from a hand-managed EC2 fleet onto EKS in waves, using a dual-ingress pattern and per-service canaries. Zero planned maintenance windows. Downtime incidents dropped 40% over the following six months. Cloud spend dropped 25% once Karpenter and savings plans kicked in. The hard parts were not technical.

Where we started

The platform ran about thirty services across a pair of Auto Scaling Groups fronted by an Application Load Balancer. Deploys were a mix of Jenkins jobs and ssh. Config lived in three places. Secrets lived in four. The AMI was baked with Packer, which was the one piece of the system I liked.

The fleet had been growing for about three years and it was becoming the main source of incidents. Most of them were some combination of:

  • Deploys that worked on one node and not another because of drift.
  • Alerts that fired on machine health rather than service health.
  • Manual cleanup between deploys, which we kept forgetting to do.
  • Autoscaling that was tuned to CPU and couldn’t react fast enough to request-rate spikes.

I’d been running a small EKS cluster for internal tools for about six months and I knew the shape of the thing. The pitch I made to the CTO was not “Kubernetes is better”. The pitch was: we cannot keep paging on the same three things every week, and a platform rebuild is the cheapest way to break that loop.

The architecture we moved to

The ALB sat unchanged in front of the world. Each service got two target groups, one pointing at the legacy fleet and one pointing at a matching Kubernetes Service (via the AWS Load Balancer Controller and a TargetGroupBinding). The shift was controlled by the ALB listener’s weighted forward action. On day one, every new target group was 100% legacy and 0% EKS. On any given day of the migration, exactly one service was in motion.

Why a single ALB and not a separate one for EKS? Because if I’d stood up a new ALB, I’d have had to do a DNS cutover per service, and DNS TTLs are not a thing you want on the critical path during a payments migration. Same ALB, weighted targets, instant rollback. This turned out to matter, see below.

The Terraform layout

I broke the infrastructure into three layers, each its own state file, with explicit read-only data flow between them.

terraform/
terraform/
├── 00-network/           # VPC, subnets, NAT, route tables
├── 10-cluster/           # EKS control plane, IRSA, addons, Karpenter
├── 20-platform/          # cluster addons: ArgoCD, external-dns, cert-manager,
│                         #                  metrics-server, aws-load-balancer-controller
├── 30-data/              # RDS, ElastiCache, MSK (shared by old and new)
└── modules/
    ├── eks-service/      # reusable: namespace, IAM role, TargetGroupBinding
    ├── karpenter-pool/   # NodePool per workload class
    └── alb-weighted/     # weighted listener + two target groups

The one I still feel good about is alb-weighted. It’s the piece that let us do the actual migration, and it’s thirty lines of HCL.

modules/alb-weighted/main.tf (abridged)
resource "aws_lb_listener_rule" "svc" {
  listener_arn = var.listener_arn
  priority     = var.priority

  action {
    type = "forward"
    forward {
      target_group {
        arn    = aws_lb_target_group.legacy.arn
        weight = var.legacy_weight
      }
      target_group {
        arn    = aws_lb_target_group.eks.arn
        weight = var.eks_weight
      }
      stickiness {
        enabled  = true
        duration = 60
      }
    }
  }

  condition {
    host_header {
      values = [var.host]
    }
  }
}

Each service’s weights lived in a tiny tfvars file in its own repo. Bumping a service from 10% to 25% was a PR with a one-line diff. That became important.

The Karpenter NodePools

I didn’t want to tune a single node group for every workload, so we set up three NodePools by workload class. General services got on-demand m6i.large and m6i.xlarge. Worker services that could tolerate interruption got spot. A tiny pool stayed on c7i on-demand for the two latency-sensitive paths where we didn’t want any chance of a cold start.

k8s/karpenter/nodepool-general.yaml
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general
spec:
  template:
    spec:
      requirements:
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: karpenter.k8s.aws/instance-family
          operator: In
          values: ["m6i", "m7i"]
        - key: karpenter.k8s.aws/instance-size
          operator: In
          values: ["large", "xlarge", "2xlarge"]
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: general
      expireAfter: 168h   # recycle weekly
  limits:
    cpu: "200"
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
Why expireAfter: 168h? Because long-lived nodes collect weird state. Kernel logs grow. Node-local caches drift. The cheapest way to avoid a bad-node mystery in six months is to force a rolling replacement every week. Karpenter handles it as a normal disruption and the HPA picks up the slack.

The ArgoCD side

I went with the App-of-Apps pattern because the team was new to GitOps and I wanted them to be able to see all of the cluster state by looking at one repo and one Argo UI. The root app lived in gitops/root-app.yaml and pointed at gitops/applications/, which contained one Application per service.

gitops/applications/payments-api.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: payments-api
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: platform
  source:
    repoURL: https://github.com/ORG/payments-api
    path: deploy/helm
    targetRevision: main
    helm:
      valueFiles:
        - values.yaml
        - values.prod.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: payments
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    retry:
      limit: 5
      backoff:
        duration: 10s
        factor: 2
        maxDuration: 3m
    syncOptions:
      - CreateNamespace=true
      - PrunePropagationPolicy=foreground

The eight-week timeline

  • Week 0 EKS cluster up via Terraform. Platform addons installed. Nothing routed yet. I spent the whole week breaking it and putting it back together in the staging account.
  • Week 1 First “easy” service on at 5% weight. An idempotent webhook dispatcher. Zero traffic-shaping drama. Good morale.
  • Week 2 Four more services at 25–50%. Discovered that one service had a hidden dependency on a specific EC2 metadata endpoint it used for logging. Fixed it and added it to the pre-flight checklist.
  • Week 3 Promoted the first batch to 100%. Turned off the legacy target group for those services. Small confidence win.
  • Week 4 The DNS/TTL incident. Described below. Migration paused for 36 hours.
  • Week 5 Resumed with three more services. Built a new pre-flight that checked for hard-coded AZ pinning.
  • Week 6 Payments API on at 10% for six hours, then 25%, then 50% overnight. This was the one that mattered. No drama.
  • Week 7 Rest of the catalog. A long-tail of small services, batched two or three per day.
  • Week 8 Last three services. ALB target groups for legacy removed. Packer AMI deleted. Jenkins box scheduled for retirement.

The thing that almost sank us

In week four I hit a problem I should have seen coming. One of the services was a worker that talked to a third-party payments processor over a long-lived TCP connection. The worker had been running the same binary on EC2 for two years. On EKS, it kept getting rescheduled by cluster events (normal pod eviction, node consolidation, that kind of thing). The third-party’s API enforced a tight rate limit on new TCP handshakes from a given source IP.

We didn’t see it right away because the legacy and EKS workers were sharing traffic. Half the handshakes came from the NAT gateway, half from the legacy fleet’s Elastic IPs. The rate limit only tripped once we started shifting more weight. When it tripped, we got back a bunch of 429s that the worker mis-handled as transient, and it retried in a tight loop.

I caught it at about 2:30am, rolled the service back to 0% on EKS, and spent the next day: (1) adding proper backoff in the worker, (2) moving the worker’s egress through a dedicated NAT with a sticky source IP, and (3) adding a pre-flight item for “any service that holds a long-lived connection to a third party gets its own egress path”.

The lesson, if there is one: it wasn’t a Kubernetes problem. It was a “we didn’t fully enumerate this service’s non-obvious dependencies” problem. Kubernetes just made the symptom obvious. Most migrations break this way. If I did it again I’d spend the first week writing a dependency inventory before I wrote any YAML.

What I’d do differently

  • Start with the dependency inventory. See above. A week of interviews with service owners would have saved us a bad night.
  • Write the alb-weighted module on day one, not week two. I hand-rolled the first couple of cutovers and regretted it.
  • Kill the legacy target groups aggressively. I left a couple of them around “just in case” for two weeks longer than necessary, and they drifted. If you’re not going to use them, delete them.
  • Don’t write a runbook for the migration itself. Write the runbook for the post-migration platform from day one. You’ll need it almost immediately.

What happened after

Six months after the cutover, incident count per week was about 40% down. Most of the ones we still had were genuine bugs, not infrastructure flakes. Cloud cost dropped by 25% over the next quarter, mostly from Karpenter consolidation and a pair of 1-year savings plans. On-call got less loud. The on-call handover meeting became fifteen minutes instead of forty-five, because the dashboard told a consistent story.

The best compliment I got was from a backend engineer who said, about two months in, “I don’t think about deploys anymore.” That was the whole point.
