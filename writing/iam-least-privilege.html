IAM least privilege without losing your weekend

A practical workflow for taking a wide-open starter IAM policy and ratcheting it down to something tight. No guessing, no theatre, no weekend lost to policy simulation.

TL;DR

Start permissive in staging. Let the workload run for a week. Ask IAM Access Analyzer for the actual actions used. Generate a tight policy from that. Review, trim, apply. Repeat for each workload. Total time per service: about 30 minutes, most of it waiting.

The wrong way

The way most people do this: stare at the AWS service API reference, try to guess which actions the workload needs, write a policy, apply it, watch things break, loosen it, watch things break differently. Lose a weekend. End up with a policy that’s still too wide because every time you got tired you added a *.

The way I do it

  1. Start permissive in staging. Give the service role the AWS-managed policy closest to what it needs, or a scoped * on one service. Let it run.
  2. Let the real usage collect. Give it at least a week, ideally a full on-call rotation so rare code paths exercise.
  3. Ask IAM Access Analyzer to generate a policy from usage. This is the key step and most people don’t know it exists.
  4. Review the generated policy by hand. Trim conditions. Add resource constraints. Add a Deny for anything destructive that you don’t want, even if the current workload doesn’t use it.
  5. Apply to staging, watch for errors, then promote to prod.

Step 3, concretely

bash
# Kick off policy generation for a role based on its last 7 days of CloudTrail activity.
aws accessanalyzer start-policy-generation \
  --policy-generation-details '{
    "principalArn": "arn:aws:iam::111111111111:role/payments-worker"
  }' \
  --region eu-west-1

# Returns a jobId. Poll until it's done.
aws accessanalyzer get-generated-policy \
  --job-id <jobId> \
  --include-resource-placeholders \
  --region eu-west-1

The generated policy is JSON, scoped to the actions Access Analyzer actually saw being called. It’ll include resource placeholders like <BucketName> that you replace with the real ARNs. It’s not perfect, and you should not just apply it blindly, but it’s a better starting point than a blank policy editor.

What I always add by hand

1. Resource constraints

The generated policy will often say “this role used s3:GetObject” without specifying which bucket. Scope it to the buckets that actually matter.

payments-worker policy (excerpt)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadConfigFromS3",
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": [
        "arn:aws:s3:::zeeh-payments-config-prod/*"
      ]
    },
    {
      "Sid": "WriteLedgerEvents",
      "Effect": "Allow",
      "Action": ["s3:PutObject"],
      "Resource": [
        "arn:aws:s3:::zeeh-payments-ledger-prod/events/*"
      ],
      "Condition": {
        "StringEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}

2. A deny on anything dangerous

Even if the workload doesn’t use it, I explicitly deny a handful of destructive actions. This is belt-and-braces: if a future engineer accidentally adds a permissive statement, this deny catches it. 

{
  "Sid": "DenyDestructive",
  "Effect": "Deny",
  "Action": [
    "s3:DeleteBucket",
    "kms:ScheduleKeyDeletion",
    "iam:DeleteRole",
    "iam:DeleteUser"
  ],
  "Resource": "*"
}

3. A source-IP or VPC endpoint condition, when applicable

For roles assumed by compute inside a VPC, I pin the trust policy to the VPC endpoint. Even if the credentials leak, they’re useless outside the VPC. 

"Condition": {
  "StringEquals": {
    "aws:SourceVpce": "vpce-0a1b2c3d4e5f"
  }
}

IRSA specifically

For Kubernetes workloads on EKS I use IRSA (IAM Roles for Service Accounts). Each workload gets its own role, trusted by its own service account in its own namespace. No node-level credentials.

terraform/modules/eks-service/iam.tf
data "aws_iam_policy_document" "assume_role" {
  statement {
    effect  = "Allow"
    actions = ["sts:AssumeRoleWithWebIdentity"]
    principals {
      type        = "Federated"
      identifiers = [var.oidc_provider_arn]
    }
    condition {
      test     = "StringEquals"
      variable = "${var.oidc_issuer_host}:sub"
      values   = ["system:serviceaccount:${var.namespace}:${var.service_account}"]
    }
    condition {
      test     = "StringEquals"
      variable = "${var.oidc_issuer_host}:aud"
      values   = ["sts.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "this" {
  name               = "${var.namespace}-${var.service_account}"
  assume_role_policy = data.aws_iam_policy_document.assume_role.json
}

How I verify it

Before I apply a tightened policy to production, I run one last sanity check: the IAM policy simulator, pointed at the specific actions I expect the workload to need. Not exhaustive, but catches the obvious misses. 

aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::111111111111:role/payments-worker \
  --action-names s3:GetObject s3:PutObject kms:Decrypt sqs:ReceiveMessage \
  --resource-arns arn:aws:s3:::zeeh-payments-ledger-prod/events/foo \
                  arn:aws:kms:eu-west-1:111111111111:key/abcd-1234 \
  --output table

Then I deploy to staging, watch CloudWatch logs for AccessDenied for an hour, and if it’s clean I promote to prod.

The audit cadence

Policies drift. New features get added, someone adds a statement to unblock themselves, the original tight policy loosens. I run an audit quarterly:

  1. List every custom IAM role we own.
  2. For each role, run get-generated-policy again.
  3. Diff the new generated policy against the current applied one.
  4. Anywhere the applied policy grants more than the generated one needs, it’s a candidate for trimming.
  5. Anywhere the applied policy is missing something the role is actively using, I’m about to cause an incident. Leave it alone.

Most quarters I find two or three candidates for trimming and one drift-in-the-wrong-direction that reveals a change nobody documented. Cheap insurance.
