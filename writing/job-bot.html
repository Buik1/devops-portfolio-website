Job Bot: a tiny personal ops project for finding remote roles

A small Node + SQLite tool I built for myself. Pulls remote DevOps listings from half a dozen boards, scores them against my profile, and drafts tailored CVs with Gemini. Nothing fancy. I use it every morning.

TL;DR

Daily ingest from five public job sources into a local SQLite db, keyword + regex scoring with a negative list, a small Express dashboard for triage, and a Gemini-powered tailoring step that drafts a CV and cover letter per job from a single profile.json. Took a weekend. Then another weekend. Then a third.

The why

I was in a job search. The usual suspects (LinkedIn, Indeed, the big boards) were full of noise. I kept seeing the same roles cross-posted, most of them not remote, most of them junior or wrong stack. I wanted a filtered daily list of senior DevOps / SRE / platform remote roles, drawn from sources where the data is actually structured.

I also wanted to stop rewriting the same CV six different ways. A reasonable LLM call per job, using a single source-of-truth profile, would save me real time if the output was any good.

Architecture (such as it is)

Sources

  • RemoteOK · JSON feed, very clean
  • Remotive · JSON API
  • WeWorkRemotely · RSS, some HTML cleanup needed
  • Greenhouse boards · public per-company JSON at boards-api.greenhouse.io/v1/boards/<slug>/jobs. I maintain a list of ~40 infra-relevant companies.
  • Lever boards · public per-company JSON at api.lever.co/v0/postings/<slug>?mode=json. Another ~24 companies.

Total fetched per run: about 4,500 listings. After dedupe, remote filtering, exclude keywords, and scoring: about 600 real candidates. After I click through them and mark most as “skip”: about five I actually want to apply to per week.

Scoring, plainly

No ML. I don’t need it here. A keyword-based score with three buckets: must-have keywords (each worth +10), nice-to-haves (+2), and excludes (-100, effectively a kill). Then bonuses: the role is clearly remote (+5), the title has “senior” or “staff” (+5), the location mentions a friendly region like EMEA or Worldwide (+3). Penalty for US-only remote (-30) because most US companies won’t take a Nigerian contractor on a W2.

filter.js (excerpt)
function scoreJob(job) {
  const hay = lower(`${job.title} ${job.company} ${job.description}`);
  let score = 0;
  const matched = new Set();

  for (const k of config.mustKeywords) {
    if (hay.includes(k)) { score += 10; matched.add(k); }
  }
  for (const k of config.niceKeywords) {
    if (hay.includes(k)) { score += 2; matched.add(k); }
  }
  for (const k of config.excludeKeywords) {
    if (hay.includes(k)) { score -= 100; matched.add('!' + k); }
  }
  if (isRemote(job))           score += 5;
  if (/senior|staff/i.test(job.title)) score += 5;
  if (isUSOnly(job))           score -= 30;

  return { score, matched_terms: [...matched], passes: score >= 0 };
}

Tailoring (the Gemini bit)

Single API call per job. Returns a strict JSON blob with { cv, cover_letter, why_fit, answers }. The prompt pulls from profile.json and faq.json, so I never have to edit the prompt; I just keep the profile accurate.

The tricky part turned out to be making the output sound human. The first draft of the prompt used defaults and produced textbook LLM prose: “leveraged” this, “passionate about” that. Real recruiters now actively reject that. I rewrote the prompt to include a long list of banned phrases, a target register (“tired, smart, mid-career engineer”), and a self-review step. Temperature 0.85. The results are noticeably more natural.

tailor.js (excerpt)
const body = {
  contents: [{ role: 'user', parts: [{ text: prompt }] }],
  generationConfig: {
    temperature: 0.85,
    topP: 0.95,
    maxOutputTokens: 16384,
    responseMimeType: 'application/json',
  },
};

The dashboard

Fifty lines of Express, one index.html, no framework. Lists jobs filtered by status and minimum score, with buttons for “interested / skip / applied”. A ✨ Tailor button per job runs the Gemini call and pops a modal with the draft CV, cover letter, FAQ answers, and a link to a print-to-PDF version of each.

Dedup works across runs by (source, external_id). Jobs I’ve already triaged don’t come back.

Stack

Runtime
Node.js, no build step
DB
SQLite via better-sqlite3 (synchronous, tiny, perfect for local)
HTTP
axios for outbound, express for the dashboard
Parsing
cheerio where needed, regex where possible
LLM
Gemini 2.5 Flash (free tier, generous, no credit card)
Scheduling
Windows Task Scheduler running npm run ingest daily at 8am

What went wrong

  • I built ingesters before filters. I ended up with 4,500 listings in SQLite and no way to find the good ones. Should have built the filter first with 100 hand-picked examples.
  • My first prompt was too strict. Temperature 0.4, “be professional”, result: generic LLM sludge. Loosened the temperature, added the voice rules, much better.
  • I used gemini-1.5-flash. It was deprecated before I finished building. Swapped to 2.5-flash, five-minute fix, but a good reminder that pinning to model versions matters.
  • First token budget was 4096. Some CVs got truncated mid-JSON. Bumped to 16k.

What I’d add if I kept going

  • A “watchlist” feature: if a specific company posts anything matching my filter, page me.
  • Auto-submit for Greenhouse/Lever/Ashby forms, behind an explicit per-job approval. Puppeteer, careful rate limiting.
  • A local embeddings store so the filter can find jobs that don’t match on keywords but match on intent.
  • A nicer charts view: which companies post most, which days, what’s the median score.

I’m not sure I’ll build those. The version I have does what I need. The thing about personal tools is that you should stop when they’re useful, not when they’re finished.
