Picking an observability stack for a team of eight

We ran Datadog and Prometheus plus Grafana plus Loki side by side for six weeks. Here’s what we kept, what we cut, and the handful of dashboards that ended up actually getting looked at.

TL;DR

We landed on Prometheus + Grafana + Loki for metrics and logs (open source), with Datadog kept only for APM and a small amount of synthetic monitoring. The deciding factor was not the tooling, it was the cost trajectory and the fact that three engineers already knew PromQL. Most of the dashboards we built never got opened. The four that did are listed below.

Why this even came up

At the start of 2023 we were on a full Datadog deployment that had grown organically. Metrics, logs, APM, RUM, synthetics, everything. It worked. The UX was good. And it was becoming a serious line item on the AWS bill’s more expensive cousin, the vendor invoice.

The CFO wasn’t asking me to cut it yet, but I could see the trajectory. Log volume was doubling every six months as the platform grew. I did the math and at our current growth we’d be spending more on observability than on compute within 18 months. That’s not automatically wrong, but it was worth a test.

The comparison, honestly

I set up a parallel stack alongside Datadog and ran both for six weeks on production. Not a benchmark, a lived-in test. The team used both for real work. Here’s what I concluded.

Datadog

  • Wins: correlated metrics/logs/traces without configuration, great APM UI, Watchdog was occasionally uncanny at catching regressions.
  • Loses: cost per ingested log line, cost per custom metric, cost per host. The unit economics do not scale down gracefully.
  • Learning curve: low.

Prometheus + Grafana + Loki (kube-prometheus-stack + Loki stack)

  • Wins: flat-rate cost (you pay for the storage and the compute, not the cardinality), PromQL is a real query language, alerting rules in YAML that live next to the code.
  • Loses: you own the stack. High cardinality will bite you. Loki’s query UX is rougher than Datadog log search if you don’t structure your logs well.
  • Learning curve: medium. PromQL is easy to misuse if you haven’t written it before.

What we kept and what we cut

ThingKept onWhy
MetricsPrometheus + GrafanaFlat-rate, flexible, already bundled with kube-prometheus-stack. Storage on S3 via Thanos.
LogsLokiCheap and the query language is close enough to PromQL that the team picked it up.
APM / tracesDatadogOSS tracing is real (Tempo, Jaeger) but our team didn’t have the bandwidth to run two new things at once. Kept DD for APM only.
SyntheticsDatadogTwenty checks, trivial cost. Not worth replacing.
AlertingAlertmanager + PagerDutyAlerts-as-code in the gitops repo, versioned with the code they alert on.
DashboardsGrafanaJSON dashboards checked into the repo, rendered from Jsonnet.

The stack, in Terraform and YAML

I installed kube-prometheus-stack via its Helm chart, with Thanos sidecars shipping blocks to S3 for long-term storage. Loki got its own install with the chunks bucket on S3 as well. Both of them run in a dedicated observability namespace with their own NodePool.

helm/values/kube-prometheus-stack.yaml (excerpt)
prometheus:
  prometheusSpec:
    retention: 10d
    retentionSize: 45GB
    thanos:
      baseImage: quay.io/thanos/thanos
      version: v0.34.1
      objectStorageConfig:
        existingSecret:
          name: thanos-objstore
          key:  objstore.yml
    resources:
      requests: { cpu: 500m, memory: 4Gi }
      limits:   { cpu: "2",  memory: 8Gi }
    externalLabels:
      cluster: prod
      region:  eu-west-1
    additionalScrapeConfigs:
      - job_name: karpenter
        kubernetes_sd_configs:
          - role: pod
            namespaces: { names: [karpenter] }

The four dashboards that actually get used

I built about 20 dashboards in the first month. Grafana’s sharing link tells you which ones get opened. After three months, four of them were doing all the work. I deleted most of the rest because unused dashboards rot.

1. The payments SLO board

One board, one service, two panels: availability SLO burn rate (1h and 6h windows) and p99 latency against the SLO ceiling. That’s it. This is the board I open first when PagerDuty wakes me.

prometheus/rules/slo-payments.yml
- record: slo:payments_availability:ratio_rate1h
  expr: |
    sum(rate(http_requests_total{job="payments-api",status!~"5.."}[1h]))
    /
    sum(rate(http_requests_total{job="payments-api"}[1h]))

- record: slo:payments_availability:error_budget_burn_rate1h
  expr: |
    (1 - slo:payments_availability:ratio_rate1h) / (1 - 0.999)

2. The cluster-wide RED board

Rate, errors, duration, grouped by Kubernetes namespace and workload. One glance tells me whether any service is misbehaving and where. Built from the istio_requests_total equivalent, in our case the AWS Load Balancer Controller metrics plus a middleware in our services that emits http_requests_total with consistent labels.

3. The Karpenter / cluster capacity board

Node count, pending pods over time, % allocation per NodePool, nodes per instance family. This was the board that would have saved us 18 minutes if I’d had it before the 2am incident.

4. The cost board

Yes, a cost dashboard in Grafana. I pull AWS Cost Explorer data into Prometheus via a small exporter that runs hourly. It’s not precise (Cost Explorer lags by a day), but it’s close enough to spot something moving in the wrong direction. The team checks it on Mondays.

Alerting philosophy

I wrote out the alerting rules in the gitops repo and made them code reviewed. The rule I kept coming back to: every alert must be actionable and must have a runbook link. If an alert fires and the on-call can’t either fix it, escalate it, or explicitly acknowledge it as a known behaviour, the alert itself is the bug.

prometheus/rules/alerts.yml (excerpt)
groups:
- name: payments-api
  rules:
  - alert: PaymentsAPIHighErrorRate
    expr: |
      (
        sum(rate(http_requests_total{job="payments-api",status=~"5.."}[5m]))
        /
        sum(rate(http_requests_total{job="payments-api"}[5m]))
      ) > 0.02
    for: 5m
    labels: { severity: page }
    annotations:
      summary: "payments-api error rate > 2% for 5m"
      runbook: "https://runbooks/payments-api/high-error-rate"
      dashboard: "https://grafana/d/payments-slo"

  - alert: PaymentsAPIP99High
    expr: histogram_quantile(0.99, sum by (le) (rate(http_request_duration_seconds_bucket{job="payments-api"}[5m]))) > 0.5
    for: 10m
    labels: { severity: page }
    annotations:
      summary: "payments-api p99 latency above 500ms for 10m"
      runbook: "https://runbooks/payments-api/high-latency"

The cost picture, after

Observability spend dropped by about 55% total. Datadog stayed as a line item (APM + synthetics), but it was a fraction of the previous bill. The self-hosted Prometheus + Loki + Thanos setup cost us maybe two steady-state nodes worth of compute and a few dollars of S3 per month. Operational burden was real but not overwhelming: roughly a day a month of maintenance, once the initial deployment was stable.

What I’d tell my past self

  • Don’t migrate logs and metrics at the same time. I did, and it made the first two weeks harder than they needed to be. Metrics first, then logs once metrics are boring.
  • Cardinality will bite you. The first week on Prometheus I had an alert that scraped a label containing a user ID. I noticed when Grafana started timing out. Good fast lesson.
  • Structured logs are the price of admission. If your services still emit printf strings, fix that before you move to Loki. LogQL can parse unstructured logs but it’s painful.
  • Keep vendor APM if you already have it. Replacing everything at once is a project. Replacing the expensive parts and keeping the good parts is a good quarter.
